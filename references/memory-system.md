# 本地记忆与记录系统

本文件定义 Life Coach skill pack 的可选本地记录层。七个 skills 负责“怎么做教练”，本文件负责“哪些用户上下文值得长期保留，下一次如何读回来”。

默认方案使用本地 Markdown 文件，不要求数据库、Notion、待办软件或 MCP。读者可以先用纯文件跑通，再让 AI 按自己的工具栈改造成外部连接，例如待办事项接滴答清单，愿景接 Notion，长期笔记接 Obsidian，本地日志继续留在文件夹中。

## 为什么需要记录系统

人生教练不是一次性问答。日程安排、拖延应对、习惯设计、复盘和人生愿景都依赖用户长期上下文：

- 用户的作息、固定承诺、精力窗口和恢复方式。
- 当前愿景、价值、角色、阶段主题和不做什么。
- 进行中的项目、等待事项、外部依赖和真实截止。
- 习惯实验、触发锚点、环境设计和失败原因。
- 反复出现的拖延模式、有效启动方式和高风险偏离。
- 每周/每天的实际反馈，而不是只看计划。

没有记录系统时，agent 很容易每次都从零开始，安排日程时忘记个人情况，复盘时看不到模式，愿景也无法逐步沉淀。

## 推荐本地目录

推荐在仓库外或仓库内被忽略的目录中保存个人数据，例如：

```text
life-coach-data/
├── profile/
│   ├── user.md
│   ├── life-compass.md
│   ├── energy-profile.md
│   └── coaching-preferences.md
├── planning/
│   ├── weekly-plan.md
│   └── daily-plan.md
├── projects/
│   └── projects.md
├── habits/
│   └── habit-tracker.md
├── reviews/
│   └── review-log.md
├── memory/
│   ├── long-term.md
│   └── daily/
│       └── YYYY-MM-DD.md
└── integrations/
    └── tools.md
```

公开仓库不应包含真实用户数据。若在仓库内创建 `life-coach-data/`，应确保它被 `.gitignore` 排除。

## 启动时读取什么

不要为了读取而读取。简单问题可以直接回答；涉及延续上下文、计划、复盘、习惯或日程时，再读取对应记录。

### 默认读取

在需要连续陪伴时，读取：

1. `profile/user.md`：稳定个人情况、作息、角色、固定限制。
2. `memory/long-term.md`：长期偏好、有效策略、反复模式。
3. `memory/daily/YYYY-MM-DD.md`：今天和昨天的状态、行动、偏离。

### 按场景读取

| 场景 | 读取 |
|---|---|
| 人生方向、价值观、阶段选择 | `profile/life-compass.md` |
| 安排今天、明天、本周 | `profile/user.md`、`profile/energy-profile.md`、`planning/weekly-plan.md`、`planning/daily-plan.md`、`projects/projects.md`、外部日历/待办摘要 |
| 项目拆解和优先级 | `projects/projects.md`、`profile/life-compass.md`、`planning/weekly-plan.md` |
| 拖延、启动困难、反复失败 | `memory/long-term.md`、`reviews/review-log.md`、相关项目和日计划 |
| 习惯设计 | `habits/habit-tracker.md`、`profile/user.md`、`profile/energy-profile.md` |
| 周/月/项目复盘 | `planning/weekly-plan.md`、`planning/daily-plan.md`、`reviews/review-log.md`、`projects/projects.md` |
| 工具接入或数据来源不清 | `integrations/tools.md` |

## 应该记录什么

### 1. 用户基本情况

记录会影响计划和教练方式的稳定事实：

- 工作日/周末作息、通勤、睡眠、运动、照护责任。
- 固定会议、家庭责任、学习窗口、可自由支配时段。
- 用户不希望被如何推动，例如讨厌打卡、害怕被催、喜欢结构化表格。
- 已确认的健康、精力或环境限制。不要诊断，不记录未经确认的医学推测。

模板：

```markdown
# 用户画像

## 基本作息

## 固定承诺

## 可用窗口

## 偏好与雷区

## 注意事项
```

### 2. 人生罗盘

记录用户确认过的方向，而不是一次性替用户写使命宣言。

- 想如何生活。
- 核心价值。
- 生活领域：身体、关系、工作与创造、学习、休息、财务、贡献等。
- 当前阶段主题。
- 本阶段不做什么。
- 正在验证的人生实验。

模板：

```markdown
# 人生罗盘

## 我想如何生活

## 核心价值

## 生活领域

| 领域 | 想靠近的方向 | 当前状态 | 本阶段关注 |
|---|---|---|---|

## 当前阶段主题

## 本阶段不做什么

## 正在进行的人生实验
```

### 3. 精力画像

记录计划时需要反复使用的容量信息：

- 高、中、低能量窗口。
- 常见消耗类型：认知疲劳、社交疲劳、情绪消耗、身体疲劳。
- 有效恢复方式和无效恢复方式。
- 透支信号和降载策略。
- 不同任务类型的大致耗时范围。

注意：单次状态不写成规律；至少多次复盘后再沉淀为长期画像。

### 4. 项目与承诺

记录多步事项，不把它们伪装成单个任务：

- 进行中项目：目标、完成标准、当前里程碑、下一步、检查日期。
- 等待事项：等谁、等什么、下次检查。
- 暂停项目：暂停原因、重启条件。
- 以后想法：为什么值得保留、何时复查。
- 已完成或放弃：结果和学到什么。

### 5. 计划与日程

日程安排要记录“计划”和“实际”，否则无法校准。

日计划至少包含：

- 已知事实：当前时间、固定承诺、截止、可用窗口、用户状态。
- 今日胜利：只完成一件也值得的一项。
- 任务临时画像：预计耗时、认知负荷、心理阻力、不确定性。
- 精力与恢复判断。
- 时间线或事件线安排。
- 最小版本和打乱时保留/放弃。
- 日终实际：实际耗时、开始前后感受、恢复效果、偏离原因、明天第一步。

### 6. 习惯实验

记录习惯不是为了制造道德压力，而是为了持续调参。

- 习惯名。
- 微小版本。
- 提示/锚点。
- 环境设计。
- 实验期。
- 成功计数或观察指标。
- 当前状态：实验中、调整中、暂停、归档。
- 失败原因和学到什么。

### 7. 长期记忆

`memory/long-term.md` 只放跨场景反复有用的信息：

- 核心洞察：用户对自己长期模式的确认。
- 有效策略：多次验证有效的方法。
- 无效策略：会造成反效果的方法。
- 反复模式：例如晚间低估疲惫、完美主义启动困难、过度承诺。
- 教练偏好：用户喜欢简洁、表格、直接提醒、温和提问等。

不要把一次对话里的推测、情绪或临时状态写成长期事实。

### 8. 工具与外部系统

记录用户实际使用哪些外部系统，以及 agent 如何读写它们：

```markdown
# 工具与数据来源

## 待办事项
- 当前工具：本地 Markdown / 滴答清单 / Todoist / 其他
- 读取内容：任务、截止、项目、优先级、标签
- 写入规则：确认后再写入；不自动创建大量任务

## 日历
- 当前工具：本地 Markdown / Google Calendar / Apple Calendar / 其他
- 读取内容：固定承诺、会议、出行
- 写入规则：只有用户确认时间块后再写入

## 愿景与知识库
- 当前工具：本地 Markdown / Notion / Obsidian / 其他
- 读取内容：人生罗盘、长期目标、复盘、笔记
- 写入规则：重要更新先总结给用户确认
```

## 写入规则

### 事实、假设、洞察分开

- **事实**：用户明确说过，或工具中已经存在的数据。
- **假设**：agent 为了推进计划做出的临时估计，必须标注“暂时假设”。
- **洞察**：经过对话确认的理解，例如“我发现我不是懒，而是害怕开始后做不好”。
- **规律**：多次复盘后稳定出现的模式，才能写入长期记忆。

### 写入前确认

以下内容写入前应先向用户确认：

- 人生价值、身份描述、长期愿景。
- 对用户性格、模式、关系的总结。
- 健康、心理、关系、财务等敏感信息。
- 会影响外部工具的操作，例如创建日历块、修改任务截止、归档项目。

可以直接写入的内容：

- 用户刚刚确认的计划。
- 明确的日程草案和日终实际记录。
- 用户要求记录的偏好、策略或复盘结论。

### 不记录什么

- 未经确认的心理诊断、人格标签或病理判断。
- 对他人的隐私细节，除非它直接影响用户行动且用户要求保留。
- 一次性情绪宣泄中的强烈措辞，除非用户确认它是长期模式。
- 可能伤害用户安全的敏感信息，除非有明确用途和保护方式。
- 密码、密钥、身份证件、银行卡等高敏感信息。

## 外部 MCP 和工具接入

本地 Markdown 是默认低门槛方案，不是唯一方案。读者可以让 AI 根据自己的工具栈改造 `integrations/tools.md` 和相关读写规则。

常见改造：

- 待办事项接滴答清单、Todoist、Things、Linear、Notion database。
- 日历接 Google Calendar、Apple Calendar、Outlook。
- 愿景和长期笔记接 Notion、Obsidian、Logseq。
- 复盘和日志保留本地 Markdown，方便版本管理和迁移。

接入外部 MCP 时要保留三个原则：

1. **读写分离**：读取可以更主动，写入必须更谨慎。
2. **确认后写入**：创建、修改、删除外部事项前先让用户确认。
3. **工具失败可降级**：MCP 不可用时，先输出 Markdown 草案，不阻塞教练过程。

## 日程安排的最低上下文

当用户说“帮我安排今天/明天/本周”时，至少需要这些信息：

- 固定承诺：会议、通勤、吃饭、运动、照护、睡眠。
- 外部任务：待办、截止、别人等待的事项。
- 项目状态：哪些项目本周真正需要推进。
- 当前状态：睡眠、身体、情绪、注意力、是否透支。
- 历史校准：类似任务通常耗时多久，什么时段最容易做成。
- 恢复需求：今天必须保护哪些休息和缓冲。

信息不完整时，先用明确标注的假设排一版，再只问一个最影响排程的校准问题。

## 维护节奏

- 每天：更新 `daily/YYYY-MM-DD.md` 或 `planning/daily-plan.md` 的实际记录。
- 每周：更新 `weekly-plan.md` 和 `review-log.md`。
- 每月或阶段变化：更新 `life-compass.md`、`projects.md`、`energy-profile.md`。
- 多次复盘后：把稳定模式提炼进 `memory/long-term.md`。

记录系统的目标不是让用户被数据管理，而是让下一次对话更懂上下文，让计划更贴近真实生活。
