# 习惯设计示例

本示例是公开、虚构、脱敏案例，用来展示 `habit-design` 如何把抽象愿望改成可重复发生的小行为。

## 用户输入

```text
我想养成每天读书的习惯，但总是坚持不了。刚开始会读几天，后面一忙就断了。有没有更容易坚持的方法？
```

## 推荐路由

优先使用 `habit-design`。如果用户卡在“读什么、为什么读”，可合并 `life-vision`；如果已经多次失败想总结原因，可先用 `review`。

## 回应示例

```text
“每天读书”这个愿望很好，但它现在还不是一个习惯设计。习惯需要更具体：在什么场景下，做多小的动作，完成后有什么反馈。

我们先不设计“每天读 1 小时”。先设计一个坏日子也能完成的版本。
```

## 习惯设计

```text
目标行为：
每天读书。

最小版本：
每天读 2 页，或者读 3 分钟。

锚点线索：
晚上刷牙后，把手机放到桌上，拿起床头的书。

环境设计：
- 书提前放在床头。
- 只放一本书，避免选择困难。
- 手机充电位置离床远一点。

即时奖励：
读完 2 页后，在日历上画一个小勾。

中断恢复规则：
断了不补读。第二天只读 1 页，恢复链条。
```

## 两周实验

```text
实验目标：
不是证明自己爱读书，而是验证“刷牙后读 2 页”这个设计能不能自然发生。

观察数据：
- 14 天里发生了几天？
- 最容易失败的是哪种晚上？
- 2 页是不是太多？
- 读完后更想继续，还是觉得有负担？
```

## 回应收尾

```text
先让习惯小到不需要斗志。等“刷牙后拿书”这条链路稳定了，再慢慢增加页数。真正要养成的不是每天读很多，而是让“我会打开书”这件事自动发生。
```

## 这个案例展示的能力

- 把抽象愿望改成具体行为。
- 用锚点线索绑定习惯。
- 设计坏日子也能完成的最小版本。
- 把中断恢复纳入系统，而不是靠自责重启。
