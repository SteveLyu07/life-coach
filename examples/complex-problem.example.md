# 复杂问题示例

本示例是公开、虚构、脱敏案例，用来展示 `complex-problem-solving` 如何拆分事实、解释、情绪、约束、相关人和最小实验。

## 用户输入

```text
我现在有个项目一直推进不动。老板希望快点上线，合作方一直改需求，团队里负责设计的人也很忙。我每天都在救火，但项目没有实质进展。我不知道到底是我能力不行，还是这个项目本身就不可能按时完成。
```

## 推荐路由

优先使用 `complex-problem-solving`。如果最后明确是排期和执行问题，再转 `planning`。如果用户已经明显过载，再合并 `energy-management`。

## 问题拆分示例

```text
我先把这个问题拆开。现在它不只是“项目推进不动”，而是至少有五个变量缠在一起：

1. 目标变量：
   老板要“快点上线”，但上线标准可能不清楚。

2. 需求变量：
   合作方持续改需求，说明需求冻结机制可能缺失。

3. 资源变量：
   设计同事很忙，关键依赖不可控。

4. 执行变量：
   你每天在救火，可能没有稳定的大块时间推进关键路径。

5. 自我解释变量：
   你开始把项目问题解释成“是不是我能力不行”，这会继续消耗判断力。

所以我们先不要回答“是不是我不行”。这更像是一个系统设计问题，需要先确认事实和边界。
```

## 澄清问题

```text
先回答三个问题就够了：

1. 如果只上线最小版本，必须包含哪 1-3 个功能？
2. 最近三次需求变更分别是什么？是谁提出的？有没有影响截止日期？
3. 设计资源本周实际能给多少小时？不是理想时间，是确定能给的时间。
```

## 最小实验

```text
实验名称：48 小时项目现实化

目标：
把“项目推进不动”从情绪压力变成可决策的信息。

行动：
1. 写出最小上线版本，最多 3 个核心功能。
2. 列出当前所有阻塞项，每个阻塞项标注负责人和最早可完成时间。
3. 给老板发一版现实排期，明确三个选项：
   - 保日期：砍范围。
   - 保范围：延日期。
   - 保质量：增加资源或减少变更。
4. 和合作方约定：超过某个时间点的新需求进入下一版本。

复盘：
48 小时后看项目是否从“混乱”变成“有选择”。
```

## 回应收尾

```text
这个项目现在最需要的不是你更用力，而是让现实约束显形。只要范围、日期、资源和变更规则不清楚，任何人都会像在泥地里跑步。下一步不是证明你行不行，而是把项目变成一个可以被选择和协商的结构。
```

## 这个案例展示的能力

- 把自责从问题中分离出来。
- 拆分事实、假设、约束和相关人。
- 生成可协商的决策选项。
- 用 48 小时最小实验推动复杂问题前进。
