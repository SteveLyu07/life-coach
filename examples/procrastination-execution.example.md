# 拖延应对与执行示例

本示例是公开、虚构、脱敏案例，用来展示 `procrastination-execution` 如何把拖延拆成具体阻力，并帮助用户进入第一步行动。

## 用户输入

```text
我这周必须写一份方案，但已经拖了三天了。每次打开文档就想去刷手机。我知道它重要，但一想到要写得很完整就很烦，也很怕写出来很烂。
```

## 推荐路由

优先使用 `procrastination-execution`。如果用户主要问题是疲惫和睡眠不足，再合并 `energy-management`。如果任务其实不清楚，再合并 `planning`。

## 回应示例

```text
我先不把这理解成懒。你这里至少有三种阻力叠在一起：

1. 任务太大：你面对的是“写完整方案”，不是一个能立刻开始的动作。
2. 标准太高：你一开始就要求它像最终稿，所以大脑会自动逃开。
3. 评价压力：你怕写得烂，于是刷手机变成短期止痛。

所以现在的目标不是“写好方案”，而是让文档从 0 变成 1。我们先做一个不会吓到大脑的版本。
```

## 启动设计

```text
第一步只做 8 分钟：

打开文档，写下 5 个标题：

1. 背景
2. 当前问题
3. 目标
4. 方案选项
5. 下一步

要求：
- 每个标题下面只写一句话。
- 句子可以很丑。
- 8 分钟到就停，不允许继续苛求自己。
```

## 复盘问题

```text
8 分钟后只问三个问题：

1. 我真的开始了吗？
2. 最卡的是不知道写什么，还是怕写不好？
3. 下一步能不能只补一个小段落？
```

## 回应收尾

```text
今天的胜利不是写完方案，而是打破“只能写最终稿”的幻觉。先写烂，才有东西能改。现在我们只需要一个很小的、不体面的第一版。
```

## 这个案例展示的能力

- 不把拖延道德化。
- 把拖延拆成任务大小、完美主义和评价压力。
- 用 8 分钟启动降低门槛。
- 先恢复行动，再追求质量。
