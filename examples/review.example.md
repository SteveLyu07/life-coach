# 复盘示例

本示例是公开、虚构、脱敏案例，用来展示 `review` 如何把失败感转化为事实、变量、学习和下一轮调整。

## 用户输入

```text
我这个月计划运动 12 次，结果只去了 3 次。我感觉自己又失败了。每次都是前几天很有动力，后面一忙就断掉。
```

## 推荐路由

优先使用 `review`。复盘后如果要重新设计运动习惯，再转 `habit-design`；如果主要问题是疲惫，再合并 `energy-management`。

## 回应示例

```text
我们先不急着给这个月打“失败”标签。先把它当成一次实验数据：计划是 12 次，实际完成 3 次，这说明你不是完全没动，而是当前设计只能支持 3 次左右。

复盘的目标不是证明你不行，而是看清：哪 3 次为什么发生了，另外 9 次为什么没发生。
```

## 事实重建

```text
先记录事实：

- 计划：本月运动 12 次。
- 实际：完成 3 次。
- 完成的 3 次分别在什么时间？
- 没完成的那些天，最常见的阻碍是什么？
- 是时间不够、地点太远、强度太高、没人提醒，还是下班后电量太低？
```

## 模式提取

```text
可能的发现：

1. 能完成的运动都发生在周末上午。
2. 工作日晚上计划运动基本失败。
3. 失败不是因为不想健康，而是因为运动被安排在一天电量最低的时候。

如果这个假设成立，问题就不是“我没有毅力”，而是“运动时间设计错了”。
```

## 下一轮调整

```text
下个月不要继续计划 12 次。先设计一个更小但更稳的版本：

- 每周 2 次。
- 固定在周三早上和周六上午。
- 每次最低版本是 15 分钟快走。
- 如果没去健身房，也算完成最低版本。

观察指标：
- 实际完成次数。
- 哪个时间最稳定。
- 完成后精力是上升还是下降。
```

## 这个案例展示的能力

- 把失败标签改成实验数据。
- 先找已发生的成功条件。
- 从计划失败中提取设计问题。
- 下一轮只改关键变量。
